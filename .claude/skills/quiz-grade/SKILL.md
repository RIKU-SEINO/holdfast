# quiz-grade — 学習ノートのクイズ採点

holdfast は「作りながら学ぶ」リポ。`docs/learning-notes/phase-N.md` に各 Phase の
読後クイズ①／実装後クイズ②があり、ユーザーが回答を書く。この skill はその回答を
**学習者向けに厳密採点**し、正答例とフィードバック（FB）を返す。

## 原則

- **厳密に採点する。** 甘くしない。概念の取り違え・不正確な定義・誤った因果は明確に減点し、理由を言う。
- **コードと照合する。** 実装に関する問いは、必ず該当 Phase の実装ファイルを Read し、
  実際のコード行・フィールド名と突き合わせてから採点する。記憶や推測で採点しない。
- **学習者向けに。** 点を付けるだけで終わらない。「何が惜しいか」「本質はどこか」「次の Phase の
  どの概念に繋がるか」を簡潔に添える。CLAUDE.md の進め方の約束（TS との癖の違いを指摘する等）に従う。
- **正解の押し付けより気づき。** 模範解答を出しつつ、ユーザー自身が言い直せるよう促す。

## 手順

1. **対象を特定する。** 引数や文脈から Phase 番号とクイズ①/②を判断する（不明なら聞く）。
   `docs/learning-notes/phase-N.md` を Read する。

2. **採点材料を集める。** 実装に関する問いがあれば、その Phase で扱っている実装ファイルを
   Read してコードと照合する。どのファイルが対象かは ROADMAP / CLAUDE.md の「現在地」と
   問いの内容から判断する。

3. **各問を厳密採点する。** 問いごとに：
   - 点数（例 `7/10`）と一言判定
   - **正答例**（その問いの模範解答）
   - **FB**：何が合っていて、何が惜しい/誤りか、本質と次への繋がり
   採点後、全体の合計点と「強み／最重要の弱点」を総評する。

4. **まずチャットで提示する。** この段階ではファイルを書き換えない。

5. **FB を書き込むか確認する。** 「この採点結果を `phase-N.md` に追記しますか？」と
   ユーザーに**必ず聞く**。勝手にファイルへ書かない。
   - 書く場合：各問の `> 回答：…` の直後に、次の形式で `> **正答例**：…` と
     `> **FB**（点数）：…` を blockquote で追記する。ユーザーの元の回答は消さない。
   - 書かない場合：チャットのフィードバックのみで終える。

## 書き込みフォーマット（例）

```markdown
> 回答： （ユーザーの元の回答。消さない）

> **正答例**：（模範解答）

> **FB**（7/10）：（何が惜しいか・本質・次への繋がり）
```

## やらないこと

- 確認なしにファイルへ書き込まない。
- ユーザーの元の回答を上書き・削除しない。
- 甘い採点で「だいたい合ってる」で済ませない。減点理由を必ず言語化する。
